# CONTEXT — Cheesestorm

치지직 스트리머들의 HotS 내전 결과를 기록하고 티어리스트를 보는 플랫폼.

## 용어 사전

### 내전 (In-house match)

스트리머 커뮤니티 내부에서 진행하는 팀 대전. 외부 래더 경기와 구분됨. 이 플랫폼이 기록하는 유일한 경기 유형.

### 스트리머 (Streamer)

내전에 참가하는 플레이어. `name`(표시명), `chzzkId`, `accountLevel`(계정레벨), `gameNames`(인게임 이름 목록), `profileImageUrl`(치지직 프로필 사진) 보유. 치지직 플랫폼 소속임을 전제. 롤은 저장 필드가 아니라 내전 기록에서 파생. 프로필 사진은 육각형 아바타로 표시하며 없으면 닉네임 이니셜로 폴백.

### 인게임 이름 (Game name)

HotS 스코어 화면에서 영웅 아이콘 아래 표시되는 배틀태그. 스트리머의 표시명·`chzzkId`와 별개의 식별자이며, 부캐·닉변으로 한 스트리머가 여럿 가질 수 있어 `gameNames` 배열로 보관. 스크린샷에서 추출한 이름을 스트리머에 매칭하는 기준이 된다.

### 미매칭 (Unmatched)

스크린샷에서 추출한 인게임 이름이 어느 스트리머의 `gameNames`와도 일치하지 않은 상태. 입력자가 수동으로 스트리머를 지정하면 그 이름이 해당 스트리머의 `gameNames`에 추가되어 다음부터 자동 매칭된다(자가 학습).

### 롤 (Role)

스트리머의 주 포지션. 값: `탱커` · `투사` · `암살자` · `지원가` · `전문가`. 수동 입력 없음 — 내전 기록에서 가장 많이 플레이한 영웅의 역할군으로 파생(동률이면 최근 경기 우선). 기록이 없으면 미표시.
_Avoid_: 수동 롤 지정

### 영웅 (Hero)

HotS 게임 내 캐릭터. 각 내전에서 스트리머가 플레이한 영웅을 경기 기록에 함께 저장함.

### 경기 (Match)

한 번의 내전. 블루팀과 레드팀 각 5인, 승자, 맵, 경기시간 포함. 각 팀원은 `[스트리머명, 영웅명]` 쌍으로 기록. 같은 멤버 구성으로 하루에 여러 번 치러질 수 있어, 경기는 (날짜 + 멤버)만으로 식별되지 않는다 — 경기시간(`dur`)이 사실상의 구분자.

### 팀 (Team: 블루 / 레드)

한 경기의 두 팀을 가르는 **임의 레이블**. `blue` · `red`는 인게임 진영(좌/우 스폰)과 무관하며 그 자체로는 아무 의미도 갖지 않는다 — 오직 "누가 같은 팀이었나"와 "어느 팀이 이겼나"만 의미가 있다. 관례상 스크린샷을 업로드한 사람의 팀이 `blue`로 들어가지만, 어떤 기능도 이 관례에 의존하지 않는다. (ADR-0002 참조) 데이터 키는 `blueTeam`/`redTeam`으로 유지하되, **사용자 화면에는 인게임 진영 오해를 막기 위해 중립 라벨(팀 1 / 팀 2)로 표시**한다.
_Avoid_: 레이블을 인게임 사이드로 해석하는 것 · UI에 "블루/레드"를 진영처럼 노출하는 것

### 인게임 진영 (In-game side, 선택)

실제 게임 내 좌(블루)·우(레드) 스폰. 팀 레이블에서 **파생되지 않고**, 알 때만 `Match.leftTeam?`(어느 레이블 팀이 좌측 스폰이었나)에 수동 기록하는 선택 필드. 맵별 진영 승률은 이 값이 채워진 경기에 한해 산출한다.

### 경기 스탯 (Match stat)

한 경기에서 플레이어 개인의 수치 기록 — 킬·어시·데스, 공성딜, 영웅딜, 힐량, 자가힐, 경험치기여. 스크린샷에서만 추출되고 스크린샷은 경기 후 소실(재취득 불가)되므로 입력 시점에 함께 저장한다. **읽는 화면이 없어 보여도 죽은 코드가 아니다** — 보존성 데이터이며 경기 상세·개인 전적에서 노출 예정.

### 티어 (Tier)

승률 기반 등급. S / A / B / C / D / unranked(미집계). 최소 3경기 충족 시 부여.

| 티어     | 승률   | 최소 경기  |
| -------- | ------ | ---------- |
| S        | 65%+   | 3경기      |
| A        | 55–65% | 3경기      |
| B        | 45–55% | 3경기      |
| C        | 35–45% | 3경기      |
| D        | ~35%   | 3경기      |
| unranked | —      | 3경기 미만 |

### 큐레이션 티어리스트 (Curated tierlist)

승률 자동 티어리스트(`/`)와 **별개로** 운영자가 스트리머를 직접 S~D에 배치하는 수동 티어리스트. 별도 페이지에 존재하며 자동 티어와 독립 — 서로 값에 영향을 주지 않는다. 자동 = 객관(승률 파생), 큐레이션 = 주관(운영자 평가). 배치 결과는 별도 저장된다.
_Avoid_: 큐레이션 배치를 승률 티어에 역반영하는 것

### 영웅 티어리스트 (Hero tierlist)

영웅별 내전 승률로 자동 산출하는 티어리스트. 스트리머 티어와 동일한 승률 기준이되 집계 단위가 스트리머가 아니라 영웅. 표본이 적은 영웅의 과대평가를 막기 위한 최소 경기 수 기준을 둔다(스트리머와 동일하게 3경기).

### 전적 (Record)

스트리머 개인의 영웅폭,누적 승/패 통계. 승률, 맵별 승률, 같은 팀이면 승률이 잘나오는 스트리머, 적으로 만나면 불리한 스트리머, 연승/연패 포함.

### 연승 / 연패 (Win streak / Loss streak)

가장 최근 경기부터 연속된 승 또는 패 수. 경기 기록에서 실시간 계산.

### 시너지 팀원 (Synergy teammate) / 천적 (Nemesis)

시너지 팀원 = 같은 팀으로 함께 뛴 경기의 승률이 높은 스트리머. 천적 = 적팀으로 만났을 때 내가 진 비율이 높은 스트리머. 둘 다 상대와 **최소 3경기 공동 출현**(같은 팀/상대 팀)을 충족한 경우만 집계하며, 미만은 데이터 부족으로 비표시. 표본 임계는 티어와 동일(3).

### 맵별 승률 (Map win rate)

스트리머가 특정 맵에서 거둔 승/패 비율. **맵당 최소 3경기** 이상에서만 산출, 미만은 데이터 부족. (인게임 진영별 승률과 별개 — 진영 승률은 `leftTeam`이 채워진 경기에 한함, ADR-0002.)

### 데이터 부족 (Insufficient data)

파생 통계의 표본이 최소 임계(대개 3경기)에 못 미친 상태. 0이나 빈칸 대신 "데이터 부족"을 명시해 "값이 0"과 "표본 없음"을 구분한다.

### MMR

현재 미구현. 향후 Elo 방식 추가 예정 (ADR-0001 참조).

## 인증 & 권한

치지직 OAuth 2.0으로 로그인. 세션은 `jose` JWT → httpOnly 쿠키. NextAuth 미사용.

### 역할 (Role)

| 역할 | 부여 조건 |
|---|---|
| `viewer` | 누구나 (로그인 불필요) |
| `streamer` | 로그인한 chzzkId가 `streamers` 컬렉션에 존재 |
| `admin` | `ADMIN_CHZZK_ID` 환경변수에 등록된 chzzkId (콤마 구분 복수 가능) |

`isStreamer = role === 'streamer' || role === 'admin'` — admin은 streamer 권한을 포함.

### 기능별 권한

| 기능 | viewer | streamer | admin |
|---|:---:|:---:|:---:|
| 전체 조회 | ✅ | ✅ | ✅ |
| 경기 입력·수정·삭제 | ✗ | ✅ | ✅ |
| 스트리머 추가·수정 | ✗ | ✅ | ✅ |
| 큐레이션 티어 편집 | ✗ | ✅ | ✅ |
| 스트리머 삭제 | ✗ | ✗ | ✅ |

### 접근 제어 구현 방식

- **미들웨어** (`middleware.ts`): `/matches/new` 경로를 viewer가 접근하면 `/?auth=required`로 리다이렉트
- **UI 가드**: 나머지 쓰기 기능(삭제 버튼 등)은 `isStreamer`/`isAdmin` 조건으로 버튼 숨김
- **안내 토스트**: `?auth=required` / `?auth=error` 쿼리 감지 시 하단 토스트 표시 후 URL 정리
- **서버 쓰기 보안** (ADR-0011): 모든 Firestore 쓰기는 Next.js API 라우트(`src/app/api/`)를 통해서만 수행됨. 각 라우트는 `requireRole(minRole)`로 JWT를 검증하고 Firebase Admin SDK로 Firestore에 씀. Firestore 규칙은 `allow write: if false`로 직접 쓰기를 전면 차단.
  - 클라이언트는 `src/lib/api-client.ts`의 fetch 래퍼를 통해 API 라우트를 호출
  - Admin SDK 초기화: `src/lib/firebase-admin.ts` (서비스 계정 환경변수: `FIREBASE_CLIENT_EMAIL`, `FIREBASE_PRIVATE_KEY`)

## 페이지 구조

- `/` — 티어리스트. 3탭: **스트리머 자동**(승률 파생) · **스트리머 큐레이션**(수동 배치) · **영웅**(영웅 승률). 스트리머 셀 클릭 시 개인 전적으로 이동
- `/matches` — 경기 목록 (타임라인 피드). 확장 시 팀별 상세 스탯 표시
- `/matches/new` — 경기 등록 (팀 라벨 중립 표시 + 선택적 인게임 진영)
- `/streamers` — 스트리머 목록(가나다순) + 추가/삭제
- `/streamers/[id]` — 개인 전적. 2탭: **개요**(요약·선호 캐릭터·시너지/천적·맵별 승률·최근 매치) · **영웅**(사용한 모든 영웅 전체 스탯 표)
