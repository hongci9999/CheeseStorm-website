# ADR-0020 — Elo 레이팅: 순수 표준 방식만 채택 (개인 보정 전면 기각)

- **날짜**: 2026-07-14
- **상태**: 채택 및 구현 완료
- **관련**: ADR-0001 (MMR 유보 — 본 ADR로 대체), ADR-0005 (자동·큐레이션 이중 티어리스트)

## 배경

자동 티어(`tier.ts`)는 승률 + 스탯 혼합 점수 기반이라 **상대가 누구였는지**를 못 본다.
내전은 팀을 의도적으로 밸런싱하므로 "약한 팀 상대로 이긴 5승"과 "강한 팀 상대로 이긴 5승"이
같은 값으로 취급된다. 상대 강도를 반영하는 지표가 필요했다 (ADR-0001이 유보했던 MMR).

Elo를 도입하되, "승패만으로는 개인 실력을 못 본다"는 문제의식에서 **개인 성과 보정**을
여러 형태로 시도했다. 전부 실패했고, 그 실패의 기록이 이 ADR의 핵심이다.

## 결정

**순수 표준 Elo만 쓴다. 개인 단위 보정은 일절 넣지 않는다.**

```ts
// src/lib/elo.ts :: calcAllElos(matches)
const K = 32;
const INITIAL_ELO = 1500;

// 양 팀 평균 Elo로 기대승률 산출
const blueTeamElo = 블루 5인 Elo 평균;
const redTeamElo  = 레드 5인 Elo 평균;
const blueExpected = 1 / (1 + 10 ** ((redTeamElo - blueTeamElo) / 400));

// 팀원 전원 동일 델타
const blueDelta = K * ((이겼으면 1 : 0) - blueExpected);
```

- 전원 1500 시작, 경기를 날짜 오름차순으로 재생하며 누적
- **같은 팀 5명은 완전히 같은 점수를 주고받는다**
- Elo 총합은 정확히 보존됨 (제로섬)
- 경기 데이터가 바뀌면 전체 재계산 (증분 갱신 안 함 — 경기 80판 규모라 비용 무시 가능)

## 기각한 대안 (실데이터 80경기 / 29명 검증)

### 1. M 보정 — 에이스/버스 배수 (커밋 f2874c0, 9bd3620에 있던 방식)

5판 이상 선수에 대해, 내 Elo가 팀 평균보다 높으면 "에이스"(승·패 모두 ×0.8),
낮으면 "버스"(승 ×0.8 / 패 ×1.2)로 델타에 배수를 곱했다. 의도는 신규 유입자의
지옥 루프 방지였으나 **정확히 반대 효과**가 났다.

- **제로섬 파괴**: 승패 배수가 비대칭이라 전체 Elo 합이 **-1,077점 증발**.
  평균이 1500 → 1463으로 디플레이션. 판수가 쌓일수록 전원 하락.
- **발동률 86%**: 800회 업데이트 중 685회 발동. 예외 규칙이 아니라 기본값이 됨.
  (기존 유닛 테스트가 합 보존을 통과한 건 시나리오가 5판 미만이라 M이 안 켜져서였다.)
- **다판수 벌점**: 버스 판정 선수는 본전 유지에 승률 60%가 필요.
  많이 뛸수록 깎인다.

| 선수     | 전적          | 승률 | 순수 표준 | M 보정 | 드레인 |
| -------- | ------------- | ---- | --------- | ------ | ------ |
| 진수     | 18W-17L       | 51%  | 1526      | 1414   | -113   |
| 끠월마녀 | 35W-33L (68판) | 51%  | 1504      | 1399   | -106   |
| 던       | 23W-37L       | 38%  | 1322      | 1159   | -163   |

- **결정적 역전**: 숙이 **0승 6패(승률 0%) → 1412**가
  끠월마녀 **35승 33패(51%, 68판) → 1399**보다 높아짐. 방어 불가.

### 2. 팀 델타 총량 고정 + 팀 내 재분배

팀이 받을 총 변화량은 표준 Elo로 확정하고, 5명이 나눠 갖는 **비율만**
개인-팀평균 차이로 조정. 제로섬은 완벽히 보존됨(편차 0.0). 그런데 순위가 붕괴했다.

- 캡틴잭 44%(24W-30L) → **1547**, 철면수심 71%(5W-2L) → 1543 보다 위
- 인간젤리 52%(31W-29L) → **1458**, 마린 27%(3W-8L) → 1434 보다 아래
- 분포 폭 압축: 1322~1743 (폭 421) → 1408~1625

### 3. 개인 기대승률 (팀 평균 대신 내 Elo vs 상대팀 평균)

고레이팅일수록 기대치가 높아 승리 보상이 작고 패배 벌점이 큰 방식.
팀 Elo의 흔한 변형이지만 여기서도 같은 병이 났다.

| 선수     | 전적          | 승률 | 팀평균 기대(채택) | 개인 기대 |
| -------- | ------------- | ---- | ----------------- | --------- |
| 베릴     | 47W-28L       | 63%  | **1743**          | 1586      |
| 던       | 23W-37L       | 38%  | **1322**          | 1435      |
| 캡틴잭   | 24W-30L       | 44%  | **1462**          | 1547      |
| 철면수심 | 5W-2L         | 71%  | **1533**          | 1543      |

분포 폭 421 → 218로 압축. 잘하는 사람은 깎이고 못하는 사람은 오르는
방향이 완벽히 일관됨 — 실력 차이가 아니라 스프링처럼 1500으로 당겨진다.

## 왜 개인 보정이 전부 실패하는가

**1. 자기 참조 피드백 루프.**
세 대안 모두 *레이팅 자체*를 *보정량의 입력*으로 쓴다. 레이팅 → 보정량 → 레이팅.
이 순환은 오차를 수렴시키지 못하고 증폭시키거나(M 보정: 다판수 드레인)
압축시킨다(재분배·개인기대: 강한 평균회귀).

**2. 관측 신호가 팀 단위뿐.**
승/패는 5명 공동 결과다. 팀 평균 레이팅이 결과를 몇 점 틀렸는지는 알 수 있지만,
그 오차가 **누구 탓인지는 이 데이터에 없다**. 개인 기여도를 Elo 차이로 쪼개는 건
측정이 아니라 가정이며, 그 가정을 레이팅으로 세우면 (1)의 순환에 빠진다.

**3. 균등 분배는 게으른 선택이 아니라, 추가 가정을 넣지 않는 유일한 선택이다.**

## 개인 실력은 그럼 어떻게 반영되나

**팀원이 매 경기 바뀌기 때문에 표준 Elo만으로 장기 수렴한다.**
팀원 구성은 장기적으로 모집단 평균에 수렴하고, 남는 신호는 자기 자신뿐이다.
내 레이팅이 실력보다 높으면 → 내가 낀 팀의 기대승률이 과대평가됨 → 기대보다 자주 짐
→ 하락. 명시적으로 팀 내 분배를 하지 않아도 자기교정이 일어난다.

베릴이 1743인 건 균등 분배에도 *불구하고*가 아니라, 균등 분배 *덕분에*
75판 동안 "베릴 낀 팀이 기대보다 자주 이겼다"가 그대로 누적된 결과다.

## 자동 티어와의 관계

**Elo는 자동 티어를 대체하지 않는다.** 별도 탭(`/` 4번째 탭)으로 병존.

- 자동 티어: 승률 + 스탯 혼합 → 전적 요약
- Elo: 승패 + 상대 강도 → 상대 강도 보정된 성적
- 큐레이션: 운영자 주관 → 팀 편성 시 실제로 참고하는 지표 (ADR-0005 유지)

`PlayerStats.eloRating`으로 노출되며 `refreshStats()`가 `streamers.eloRating`에 저장.

## 계산 과정 추적 UI

Elo는 블랙박스로 보이기 쉬워 **행 클릭 시 계산 과정을 펼쳐 보여준다**.

- `calcAllElosWithDetails()` — `calcAllElos`와 동일 로직 + 경기별 중간값 기록
- `GET /api/elo-details?streamerId=` — 선수별 변동 이력
- 표시: 번호 · 날짜 · 우리팀 Elo · 상대팀 Elo · 기대승률 · 승패 · 변화량 · 누적 Elo
- 최근 10경기만. 번호는 통산 판수 기준(오래된 경기가 낮은 번호)

**Firestore 읽기 비용 0.** API는 `getMatchesCachedServer()`(`unstable_cache`, tag `matches`)를
쓰므로 경기 데이터가 바뀔 때만 읽는다. 클릭 횟수와 무관. 클라이언트에도 세션 메모리 캐시
(`eloDetailCache`)를 둬 같은 행 재클릭 시 API 호출조차 생략.

## 보류 (YAGNI)

- **판수별 K-factor** (신규 10판 미만 K=48 등): 이건 "누가 잘했나"(귀책)가 아니라
  "이 사람 레이팅을 얼마나 믿나"(불확실성)의 문제라 자기 참조 순환이 없고,
  승/패 대칭이라 M 보정 같은 방향성 편향도 없다. **정당한 유일한 차등이지만 지금은 불필요** —
  80판 규모에선 수렴 속도보다 안정성이 중요. 신규 유입이 많아지면 도입 검토.
- **Glicko / TrueSkill**: 레이팅 불확실성(RD)까지 다루는 정식 베이지안 모델.
  표본이 수백 판대로 늘고 신규 유입이 잦아지면 재검토.
- **시즌 격리**: `matches.season` 필드 도입 시 시즌별 Elo 초기화 정책 필요. 현재 전체 = 시즌 1.
- **증분 갱신**: 매번 전체 재계산 중. 경기 수백 판 넘어가면 검토.

## 자가 검증

`src/lib/elo.test.ts` — 6 케이스 통과. 핵심은 두 개:

- **Elo 합 보존**: 경기 후 전체 합이 초기값과 일치 (오차 < 0.1). M 보정이 이걸 깼다.
- **낮은 승률이 높은 랭크가 될 수 없음**: 14승 24패(36.8%) 선수가 1500 미만.

실데이터 검증은 Firestore 덤프를 임시 테스트로 돌려 확인 후 삭제
(순위표 전체, 이상치 탐지, 대안 3종 비교). 본 ADR의 표가 그 결과다.
