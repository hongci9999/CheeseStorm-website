# ADR-0005 — 스트리머 티어리스트 이원화: 자동(승률) + 큐레이션(수동)

## 상태

채택 (accepted)

## 결정

스트리머 티어리스트를 두 개의 **독립된** 뷰로 운영한다.

- **자동 티어리스트** — 승률에서 파생(기존 `calcPlayerStats`, 티어 = S~D, 최소 3경기). 객관 지표. 손으로 못 바꾼다.
- **큐레이션 티어리스트** — 운영자가 스트리머를 직접 S~D에 배치하는 수동 티어. 별도 저장. 승률과 무관.

두 값은 서로에게 영향을 주지 않는다. 영웅 티어리스트(별건)는 자동(승률)만 둔다.

## 이유

- req(L25–26)는 "영웅 티어는 자동, 스트리머 티어는 수동 조작 가능"을 요구한다. 그런데 도메인의 「티어」는 **승률 파생**으로 정의돼 있어(CONTEXT), 자동 티어를 손으로 덮어쓰면 "티어 = 승률"이라는 핵심 불변식이 깨진다.
- 객관(승률)과 주관(운영자 평가)은 둘 다 가치가 있으나 **섞으면 둘 다 신뢰를 잃는다** — 자동값을 수동으로 바꾸면 그게 승률인지 편애인지 알 수 없다.
- 그래서 덮어쓰기(override) 대신 **분리**를 택했다. 자동은 불변식을 지키고, 큐레이션은 자유를 갖는다.

## 고려한 대안

- **(B) 자동 티어에 수동 오버라이드(핀) 레이어** — 한 화면에서 보정. 거부: 승률 불변식 오염, "왜 이 티어?" 추적 불가.
- **(C) 완전 수동 보드로 전환(승률 폐기)** — 객관 지표를 영구히 버림. 거부.
- **(D) 같은 티어 내 순서만 수동** — 표현력 부족(운영자가 티어 자체를 바꾸고 싶어함). 거부.

## 결과

- 큐레이션 배치는 별도 저장이 필요하다(예: `curatedTiers` 문서 — streamerId → 티어 매핑). 자동 경로(`calcPlayerStats`)는 건드리지 않는다.
- 메인 `/`는 자동/큐레이션/영웅 3탭으로 노출한다.
- **인증 미구현(ADR-0004) 하에서는 큐레이션도 누구나 편집 가능** — "신뢰된 운영자만 만진다"는 관례에 의존하며, 관리자 로그인이 들어와야 강제된다.
- 스트리머 삭제 시 큐레이션 배치의 고아 참조를 정리해야 한다(소프트 가드).
- 자동 티어의 부가 문구(에이펙스/APEX 등 등급 설명, req L27)는 제거한다 — 큐레이션과 톤을 맞추고 과한 수사 배제.
