# ADR-0022 — 대회 경기는 Elo 집계에서 제외

- **날짜**: 2026-07-20
- **상태**: 채택 및 구현 완료
- **관련**: ADR-0020 (Elo 순수 표준 방식만 채택)

## 배경

대회(`TOURNAMENT_TEAMS`)는 팀장 드래프트로 4개 팀이 짜이고, 그 **로스터가 대회 내내 고정**된다.
대회 경기도 내전과 같은 `matches` 컬렉션에 저장되고 `tournamentGames`에 별도 태깅만 되므로,
`calcAllElos(matches)`가 전체 경기를 받는 구조상 대회 경기가 개인 Elo에 그대로 반영되고 있었다.

## 결정

**대회로 태깅된 경기는 개인 Elo 집계에서 제외한다.**

```ts
// src/lib/elo.ts
export function excludeTournament(matches: Match[]): Match[] {
  return matches.filter((m) => !m.tournament);
}
```

`calcAllElos`와 `calcAllElosWithDetails` 양쪽이 집계 시작 전에 이 필터를 통과시킨다.
대회 경기만 치른 선수는 Elo 맵에 등록되지 않고, `calcPlayerStats`의 `?? 1500` 폴백으로
초기값을 받는다 (내전 표본이 없으므로 타당).

## 근거

### 1. 고정 로스터는 팀 내 개인 정보를 만들지 못한다

ADR-0020에 따라 팀원 5명은 **항상 동일한 델타**를 받는다. 이 설계가 내전에서 성립하는 건
매 경기 팀이 새로 짜여, 경기가 쌓이면 팀원 구성이 평균화되면서 개인 기여가 조금씩 분리되기
때문이다.

대회는 정반대다. 1팀의 5명은 대회 전 경기에서 같은 폭으로 오르내린다. 팀이 8승 2패면
가장 못한 팀원이 가장 잘한 팀원과 똑같이 오르고, 진 팀의 최고 선수가 최악의 선수와 똑같이
내려간다. **팀 내 서열에 대한 정보량은 0인데 레이팅 이동폭만 커진다.**

### 2. 반복 경기는 독립 관측이 아니다

Elo는 각 경기가 독립 시행이라는 가정 위에 있다. 고정된 네 로스터끼리의 여러 경기는
"어느 팀이 센가"라는 **관측 하나를 여러 번 세는 것**에 가깝다. 정보량 대비 레이팅이
과하게 움직인다.

### 3. 드래프트는 밸런싱의 반대다

운영자가 밸런싱하는 내전과 달리 팀장 드래프트는 강한 선수를 한쪽에 모은다.
`elo.ts`의 로지스틱 `winnerGain`은 "내전은 밸런싱돼 기대승률이 30~70%에 몰린다"는
전제로 튜닝돼 있어 대회는 그 전제를 벗어난다. (다만 이 곡선이 예상된 승리의 이득을
+5까지 감쇠시키므로 세 번째 근거는 앞의 둘보다 약하다.)

## 기각한 대안

**대회 경기를 포함하되 K를 낮춘다** — 상관된 반복 관측은 완화되지만 1번(팀 내 정보량 0)이
전혀 해결되지 않는다. 튜닝 노브만 늘어 ADR-0020의 "단순하게 유지" 기조에 어긋난다.

**대회 성적을 개인 Elo에 반영하되 별도 가중** — 위와 같은 이유. 그리고 어떤 가중치를
정당화할 근거가 없다.

## 남은 여지 — 팀 단위 Elo

고정된 네 로스터가 서로 맞붙는 구조는 Elo가 **원래 설계된 상황 그 자체**다.
대회에서 Elo가 부적절한 건 대상이 선수일 때뿐이고, **팀을 레이팅 주체로 두면 통계적으로
타당하다.** 개인 Elo에서 대회를 빼는 대신 팀 Elo를 두면 버리는 데이터 없이 지표가 하나
늘어난다. 이번 결정 범위 밖이며, 필요해지면 대회 페이지 지표로 검토한다.

## 구현 — 플래그 비정규화

`tournamentGames` 컬렉션을 읽어 거르는 대신, **경기 문서에 `tournament?: boolean`을
비정규화**했다.

홈(`app/page.tsx`)은 `'use client'`로 브라우저에서 `matches`를 직접 읽어 Elo를 계산한다.
여기서 `tournamentGames`를 추가로 읽으면 **방문자 1인당 컬렉션 읽기가 늘어난다.**
Spark 무료 플랜에 연 1~2일 천 명 규모 스파이크가 걸리는 운영 조건(문서 단위 과금)에서는
감당하기 어렵다. 경기 문서의 불리언 필드는 읽기 증가가 0이다.

두 소스가 어긋나면 Elo가 조용히 틀어지므로, 쓰기는 `linkMatchToTournament` /
`unlinkMatchFromTournament` **두 함수가 전담**한다. 두 함수 모두 태그 컬렉션과 경기 플래그를
함께 갱신하고 `scheduleRefresh()`를 호출한다 — `addMatch`가 예약한 재집계는 플래그가 쓰이기
전에 경기를 읽을 수 있어서, 플래그 확정 후 한 번 더 예약해야 한다.

주의: `unlinkMatchFromTournament`는 경기 문서의 플래그도 해제하므로 **경기 삭제보다 먼저**
불러야 한다. `DELETE /api/matches/[id]`가 이 순서를 지킨다.

기존 기록은 `scripts/migrate-match-tournament-flag.mjs`로 일회성 반영한다 (드라이런 기본).

## 영향

- 개인 Elo와 티어리스트 Elo 정렬이 내전 경기만 반영한다.
- `/api/elo-details` 상세 패널에서 대회 경기가 사라진다.
- **자동 티어(승률·스탯 혼합 점수)는 여전히 대회 경기를 포함한다.** 이번 결정은 Elo 한정이다.
  자동 티어까지 제외할지는 별도 판단이 필요하다 (자동 티어는 애초에 실력 지표가 아니라
  전적 요약으로 규정돼 있어 우선순위가 낮다).
