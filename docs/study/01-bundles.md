> 앵커 사례: [00. firebase 번들](00-case-study-firebase-bundle.md)

## 1.1 번들이란

내가 짠 코드는 파일 수십 개로 흩어져 있다. 우리 프로젝트만 해도 `src/` 아래 `.tsx`·`.ts`가 수십 개.

```
src/components/site-header.tsx
src/lib/firestore.ts
src/lib/tier.ts
src/app/page.tsx
... 수십 개
```

문제: **브라우저는 이 흩어진 파일 구조를 그대로 못 읽는다.** import로 얽힌 수십 개 파일을 사용자가 일일이 받게 하면 요청만 수십 번 — 느리다.

그래서 빌드 도구(Next.js는 내부적으로 Turbopack/webpack 사용)가 이 파일들을 **합치고 압축해서 몇 개의 큰 `.js` 덩어리**로 만든다. 이 덩어리가 **번들(bundle)**, 또는 잘게 쪼갠 단위로는 **청크(chunk)**.

우리 프로젝트 실제 빌드 결과 (이번 세션 측정):

```
총 클라이언트 JS: 약 1,060 KB / 18개 청크 파일
그중 최대 = firestore SDK 청크 271 KB
```

---

## 1.2 왜 "쪼개나" — 코드 스플리팅

번들을 **하나의 거대한 파일**로 만들 수도 있다. 근데 그러면 문제가 있다.

사용자가 `/guide`(사용방법) 페이지만 보러 왔는데, 거대 번들 하나에 경기입력·OCR·드래그앤드롭 코드까지 전부 들어있으면 → **안 쓸 코드까지 다 다운로드.** 낭비.

그래서 Next.js는 코드를 **여러 청크로 쪼갠다.** 이게 **코드 스플리팅(code splitting)**. 원칙:

> 이 페이지를 보는 데 꼭 필요한 코드만 받게 한다.

`/guide`를 열면 → 가이드에 필요한 청크만.
`/matches/new`를 열면 → 경기입력에 필요한 청크만.

이래서 "파일 1000개여도 괜찮다"는 말이 성립한다. 각 페이지는 자기 몫만 받으니까.

## 1.3 두 종류의 청크 — 페이지 청크 vs 공유 청크

여기가 핵심 개념. 청크는 크게 두 부류로 나뉜다.

### 페이지 청크 (page chunk)
그 페이지에서만 쓰는 코드.
- 예: 경기입력 폼 로직 → `/matches/new`에서만.
- 다른 페이지 열면 안 받음.

### 공유 청크 (shared chunk)
**여러 페이지가 공통으로 쓰는 코드.**
- 예: React 자체, 그리고 **모든 페이지에 나오는 헤더/푸터**.
- 매번 페이지마다 따로 받게 하면 중복이니까, 한 청크에 모아서 **사이트 들어오는 순간 한 번 받고 계속 재사용.**

왜 공유 청크가 효율적이냐: 페이지 5개가 전부 React를 쓰는데 페이지마다 React를 따로 넣으면 5배 중복. 공유 청크에 한 번만 넣으면 1번 다운로드로 끝.

## 1.4 "헤더는 왜 모든 페이지에 실리나"

이제 앵커 사례의 핵심이 설명된다.

우리 `site-header.tsx`(헤더)는 `layout.tsx`(글로벌 레이아웃)에 들어있다:

```
layout.tsx (모든 페이지를 감싸는 틀)
 ├─ <SiteHeader />   ← 모든 페이지 위에 뜨는 헤더 바
 ├─ {children}       ← 여기에 각 페이지 내용이 들어감
 └─ <SiteFooter />
```

어느 페이지를 열든 헤더는 항상 화면에 나온다. 그러니 Next.js는 헤더 코드를 **공유 청크**에 넣는다. → 사이트 들어오면 무조건 받는 코드.

**문제 발생 지점:**
헤더가 무거운 걸 import 하면(앵커 사례의 firebase SDK 270KB), 그 무게가 **공유 청크에 들어가서 모든 페이지에 강제로 실린다.**

```
헤더가 firebase SDK 끌어옴
  → 헤더는 공유 청크
    → firebase SDK가 공유 청크에 포함
      → /guide 같은 무관한 페이지도 270KB 강제 다운로드
```

그래서 헤더·레이아웃처럼 **공유 청크에 들어가는 컴포넌트의 import는 영향력이 모든 페이지로 퍼진다.** 무거운 걸 함부로 import 하면 안 되는 이유.

> 우리가 한 수정 = 헤더에서 firebase 의존을 떼어내 → firebase SDK가 공유 청크에서 빠지고 → 진짜 쓰는 페이지의 **페이지 청크**로 이동. (자세히는 [2장 import 그래프](02-import-graph.md))

## 1.5 직접 확인하기

빌드하면 청크 파일들이 `.next/static/chunks/`에 생긴다. (단, `npm run dev`가 아니라 `npm run build` 해야 함 — dev는 최적화 안 함)

```bash
# 1. 프로덕션 빌드
npm run build

# 2. 청크 크기 큰 순으로 보기 (PowerShell)
```
```powershell
$d=".next/static/chunks"
Get-ChildItem $d -Recurse -Filter *.js |
  Sort-Object Length -Descending |
  Select-Object -First 10 |
  ForEach-Object { "{0,8:N0} KB  {1}" -f ($_.Length/1KB), $_.Name }
```

큰 청크가 어떤 라이브러리인지 궁금하면 그 청크 안을 문자열 검색하면 된다 (예: `firebase`, `firestore`). 자세한 측정법은 [8장](08-measure.md).

## 1.6 정리

- **번들/청크** = 흩어진 내 코드를 합쳐 압축한 `.js` 덩어리. 브라우저가 받는 것.
- **코드 스플리팅** = "이 페이지에 필요한 것만 받게" 청크를 쪼개는 것.
- **페이지 청크** = 그 페이지 전용. **공유 청크** = 모든/여러 페이지 공통(React, 헤더 등).
- **헤더는 공유 청크** → 헤더가 import 하는 무게는 **전 페이지에 퍼진다.** ← 앵커 사례의 뿌리.
