# 사례 연구 — firebase SDK가 모든 페이지에 실린 사연

> 이 문서는 학습 경로의 **앵커(anchor) 사례**다. 0~4·8장의 개념이 이 한 사건에 전부 녹아 있다. 다른 장을 읽다가 막히면 여기로 돌아와 실물에 대입할 것.
> 관련 결정 기록: [ADR-0017](../adr/0017-firebase-config-bundle-split.md)

---

## 1. 먼저 "번들"이 뭔지

내가 짠 코드는 `.tsx`, `.ts` 파일 수십 개로 흩어져 있다. 그런데 **브라우저는 이 파일들을 그대로 못 읽는다.** 그래서 빌드(`next build`)할 때, 이 수십 개 파일을 합치고 압축해서 **몇 개의 큰 `.js` 파일**로 만든다. 이 합쳐진 결과물이 **번들(bundle)**.

사용자가 사이트에 접속하면:
1. 브라우저가 HTML을 받음
2. 그 HTML이 "이 `.js` 파일들 다운로드해" 라고 지시
3. 브라우저가 그 `.js`(번들)들을 **다운로드 → 압축 해제 → 파싱 → 실행**

핵심: **번들이 크면 이 4단계가 다 느려진다.** 다운로드도 오래 걸리고, 파싱·실행에 CPU도 더 쓴다. 특히 휴대폰은 CPU가 약해서 더 느리다. 이게 "첫 화면 뜨는 속도"에 직결된다.

---

## 2. "서버 코드"와 "브라우저 코드"는 다르다

내 코드가 전부 브라우저로 가는 건 아니다.

- **서버에서만 도는 코드** — 예: `firebase-admin`, `jose`(JWT), API 라우트. Vercel 서버에서 실행되고 결과만 브라우저로 보낸다. **브라우저는 이 코드 자체를 절대 안 받는다.** 아무리 무거워도 번들 크기에 영향 없음.
- **브라우저로 가는 코드** — 클릭·입력·애니메이션 같은 상호작용을 하려면 그 코드가 사용자 브라우저에서 직접 돌아야 한다. 번들에 포함돼 다운로드됨.

Next.js에서 파일 맨 위에 `'use client'` 적혀 있으면 = "이건 브라우저로 보내는 코드". `site-header.tsx` 맨 위에 `'use client'` 있다 → 헤더는 브라우저로 가는 코드.

---

## 3. "import"가 무게를 끌고 온다

JavaScript에서 한 파일이 다른 파일 기능을 쓰려면:

```ts
import { 무언가 } from './다른파일';
```

번들러(코드 합치는 도구)는 이걸 보고 판단한다:

> "이 파일이 `다른파일`에서 뭘 가져다 쓰네? 그럼 `다른파일`도 번들에 포함시켜야겠다."

그런데 `다른파일`이 또 맨 위에서 다른 걸 import 하면? 그것도 포함. 그게 또 import 하면? 그것도 포함. **import가 꼬리에 꼬리를 물고 줄줄이 딸려오는 게 "import 그래프(의존성 그래프)".**

비유: 친구 한 명 파티에 초대했는데, 그 친구가 "내 룸메이트도 데려갈게", 룸메이트가 "내 사촌도"... 결국 한 명 부른 게 열 명 된다.

---

## 4. 버그가 정확히 이거였다

`site-header.tsx`(헤더)는 화면에 "오프라인" 뱃지를 띄울지 정하려고 **불리언 값 딱 하나**가 필요했다:

```ts
isFirebaseConfigured  // true 아니면 false, 그게 다
```

이 값 자체는 가볍다. env 변수 하나 있는지 확인이 전부:

```ts
isFirebaseConfigured = !!process.env.NEXT_PUBLIC_FIREBASE_PROJECT_ID;
```

**문제는 이 값을 어디서 가져왔냐.** 헤더는 이걸 `firestore.ts`에서 import 했다:

```ts
import { isFirebaseConfigured } from '@/lib/firestore';  // ← 함정
```

그런데 `firestore.ts` 맨 위:

```ts
import { collection, doc, getDocs, ... } from 'firebase/firestore';  // 270KB SDK 전체
import { calcPlayerStats } from './tier';
import { calcHeroTiers } from './hero-tier';
// ... 무거운 것들 잔뜩
```

`firestore.ts`는 firebase SDK(270KB)를 줄줄이 달고 있는 무거운 파일이다.

3번 규칙대로 번들러는 판단한다:

> "헤더가 `firestore.ts`에서 값을 가져가네? 그럼 통째로 포함. 어, firebase SDK 270KB도 import 하네? 그것도 포함."

**불리언 하나 쓰려고 270KB SDK 전체가 헤더에 딸려왔다.** 친구 한 명 부르려다 270명 온 셈.

---

## 5. 왜 "헤더"라서 특히 나빴나

이 실수가 `/matches/new` 같은 한 페이지에서만 났으면 그 페이지만 무거웠을 것이다. 근데 하필 **헤더**였다.

헤더는 `layout.tsx`(글로벌 레이아웃)에 들어가서 **모든 페이지에 공통으로 나온다.** 어느 페이지를 열어도 위에 헤더 바가 뜬다.

Next.js는 모든 페이지가 공유하는 코드를 **"공유 청크(shared chunk)"** 에 모아서, 사이트 들어오면 무조건 다운로드시킨다.

결과:

> 헤더가 270KB SDK를 달고 있음 → 헤더는 공유 청크 → **모든 페이지가 270KB SDK를 강제 다운로드.**

심지어 `/guide`처럼 DB를 전혀 안 건드리는 페이지도, 쓰지도 않는 firebase SDK를 받아야 화면이 떴다.

---

## 6. 어떻게 고쳤나

원인은 "가벼운 값을 무거운 파일에서 꺼냈다". 해법: **그 값을 아무것도 안 딸려오는 깨끗한 새 파일로 옮긴다.**

새 파일 `firebase-config.ts` — import가 하나도 없다:

```ts
// firebase-config.ts — 딸려오는 거 0개
export const isFirebaseConfigured = !!process.env.NEXT_PUBLIC_FIREBASE_PROJECT_ID;
```

헤더가 firestore 대신 이 깨끗한 파일에서 가져오게 변경:

```ts
// 전: import { isFirebaseConfigured } from '@/lib/firestore';       (270KB 딸려옴)
// 후: import { isFirebaseConfigured } from '@/lib/firebase-config'; (아무것도 안 딸려옴)
```

이제 번들러 판단:

> "헤더가 `firebase-config.ts`에서 값 가져가네? 근데 이 파일은 아무것도 import 안 하네? 이 파일만 포함하면 끝."

270KB SDK는 더 이상 헤더에 안 딸려온다.

> 참고: 기존에 `firestore.ts`에서 이 값을 쓰던 코드가 안 깨지게, `firestore.ts`·`firebase.ts`도 새 파일 값을 그대로 **재수출(re-export)** 하게 해뒀다. 다리만 놔준 거라 동작은 동일.

---

## 7. 결과

firebase SDK가 "모든 페이지 공유"에서 "**진짜 쓰는 페이지만**"으로 바뀌었다:

| 페이지 | 전 | 후 | 설명 |
|--------|-----|-----|------|
| 티어리스트 `/` | 270KB | 270KB | 진짜 DB 읽음 → 필요 |
| 경기입력 `/matches/new` | 270KB | 270KB | 진짜 DB 씀 → 필요 |
| **가이드 `/guide`** | 270KB | **0** | DB 안 씀 ✅ |
| **경기상세 `/matches/[id]`** | 270KB | **0** | 서버 렌더 ✅ |
| **프로필 `/streamers/[id]`** | 270KB | **0** | 서버 렌더 ✅ |

경기 상세·프로필은 사람들이 자주 들어오는 페이지인데, 이제 첫 로딩에 270KB를 안 받는다. 특히 휴대폰·느린 인터넷에서 체감.

---

## 8. 한 줄 교훈

> **번들 무게는 "무엇을 쓰느냐"가 아니라 "어디서 import 하느냐"가 결정한다.**

가벼운 값이라도 무거운 파일에서 꺼내면 그 무게가 통째로 따라온다. 특히 헤더·레이아웃처럼 모든 페이지가 공유하는 곳에서는, 무거운 파일을 거치지 않게 각별히 조심해야 한다.

---

## 이 사례가 건드리는 개념 (다른 장으로)

- 번들이란 / 공유 청크 → **1장**
- import 그래프 → **2장**
- 왜 firebase는 트리셰이킹으로 안 빠졌나 → **3장**
- 서버 컴포넌트라 SDK 안 받는 이유(프로필·경기상세) → **4장**
- 270KB를 직접 측정한 법 → **8장**
