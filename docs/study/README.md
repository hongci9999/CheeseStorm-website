# 개발 공부 — Cheesestorm로 배우는 웹 성능

이 프로젝트(Next.js + Firebase)에서 실제로 마주친 문제를 교재 삼아, 웹 프론트엔드 성능 개념을 바닥부터 쌓는 학습 노트.

추상적 이론이 아니라 **이 코드베이스의 실제 파일·실제 결정**에 묶어서 설명한다. 각 장은 "개념 → 우리 코드에서 어디에 해당하나 → 직접 확인하는 법" 순서.

> 첫 사례(앵커): [00. firebase SDK가 모든 페이지에 실린 사연](00-case-study-firebase-bundle.md) — 0~4·8장 개념이 한 사건에 다 녹아 있음. 결정 기록은 [ADR-0017](../adr/0017-firebase-config-bundle-split.md)

---

## 학습 경로

처음이면 위에서부터 순서대로. 급하면 **0장 → 4장 → 8장**만 봐도 오늘 한 작업은 이해된다.

### 0장. 큰 그림 — 코드가 화면이 되기까지
- 내가 짠 `.tsx` 파일이 사용자 브라우저에서 도는 과정
- 빌드 → 번들 → 다운로드 → 파싱 → 실행
- "번들이 크면 왜 느린가" 한 문장 정리
- 📍 우리 코드: `next build`가 만드는 `.next/static/chunks`

### 1장. 번들과 코드 스플리팅
- **번들(bundle)** 이란 무엇인가
- **공유 청크 vs 페이지 청크** — Next.js가 "모두에게" / "이 페이지만" 나누는 원리
- 왜 헤더 코드는 모든 페이지에 실리나 (layout = 공유)
- 📍 우리 코드: `site-header.tsx`가 공유 청크인 이유

### 2장. import 그래프 — 무게가 딸려오는 원리
- `import`가 의존성을 줄줄이 끌고 오는 메커니즘
- "가벼운 값을 무거운 파일에서 꺼내면 무게가 통째로 따라온다"
- 파티 비유: 한 명 불렀는데 270명 온 사연
- 📍 우리 코드: `site-header → firestore.ts → firebase SDK 270KB` (ADR-0017의 핵심)

### 3장. 트리셰이킹 (tree-shaking)
- "안 쓰는 코드는 자동으로 빼준다"는 기술
- 언제 작동하고 **왜 우리 firebase는 안 빠졌나**
- side-effect import가 트리셰이킹을 막는 경우
- 📍 우리 코드: `firebase.ts`의 `getFirestore(app)` 최상단 실행

### 4장. 서버 컴포넌트 vs 클라이언트 컴포넌트  ⭐핵심
- `'use client'`가 정확히 무엇을 바꾸나
- 서버 컴포넌트는 왜 브라우저로 JS를 안 보내나
- **왜 프로필·경기상세 페이지는 firebase SDK를 안 받았나**
- 📍 우리 코드: `streamers/[id]/page.tsx`(서버) vs `site-header.tsx`(클라)

### 5장. SSR과 하이드레이션
- 서버가 HTML을 먼저 그려 보내는 이유 (첫 화면 속도)
- 브라우저가 JS로 페이지를 "살아나게" 하는 과정 = 하이드레이션
- 서버/클라 경계에서 자주 나는 버그
- 📍 우리 코드: `firestore.server.ts`(서버 전용)와 `firestore.ts`(클라) 분리 이유

### 6장. 압축과 전송 크기
- raw 270KB ≠ 실제 네트워크 전송량
- minify(코드 줄이기) vs gzip/brotli(전송 압축)
- "측정값 270KB"를 어떻게 읽어야 하나
- 📍 우리 코드: 빌드 산출물 크기 측정 시 주의점

### 7장. lazy import (동적 import)
- `import()` 문법으로 "필요할 때만 늦게 불러오기"
- 번들을 더 잘게 쪼개는 또 다른 무기
- 언제 쓰고 언제 과한가
- 📍 우리 코드: OCR/스크린샷 파싱 같은 무거운 기능 후보

### 8장. 직접 측정하기  ⭐실전
- 빌드 후 어느 페이지가 무거운지 혼자 확인하는 법
- 청크 크기 보는 명령, 라우트별 매핑 읽는 법
- ADR-0017에서 "전/후 표"를 어떻게 만들었나
- 📍 우리 코드: `.next/static/chunks`, `page_client-reference-manifest.js`

### 9장. 진짜 병목은 따로 있다 — Firestore 읽기
- 번들보다 더 큰 리스크: Spark 무료 플랜 읽기 한도
- 천명 동시 접속 스파이크 시 무슨 일이 나나
- 캐싱·서버 컴포넌트가 읽기 횟수를 줄이는 원리
- 📍 우리 문서: [firestore-reads-analysis.md](../firestore-reads-analysis.md), ADR-0006
